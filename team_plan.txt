JobLink Hub
Team Work Breakdown Plan
4 Members  |  2 Backend  ·  2 Frontend  |  7 Weeks
1. Team Overview
JobLink Hub will be built by a team of four developers split evenly between backend and frontend responsibilities. Each developer owns a clearly defined domain so work can happen in parallel without blocking each other.
Backend Dev 1  (BE1)  — Database Design, Core API & Opportunity Module
Backend Dev 2  (BE2)  — Authentication, Services Layer & Notifications
Frontend Dev 1  (FE1)  — Public-Facing Pages, Job Search & Candidate Profiles
Frontend Dev 2  (FE2)  — Dashboards, Shared UI Components & Admin Panel
2. Individual Responsibilities
Backend Dev 1 — Database, Core API & Opportunities
BE1 owns the foundation that the entire project builds on — the database schema and the Opportunities (Jobs) module.
Design and set up the full database schema (Users, Profiles, Opportunities, Applications, Skills, Notifications)
Write and manage all Entity Framework Core migrations
Seed the database with test categories, skills, and sample data
Build the AppDbContext and all repository classes
Implement all Opportunity API endpoints: list, filter, search, create, update, delete, view tracking
Implement the Applications API endpoints: submit, view, withdraw, status updates
Implement the Saved Jobs / Saved Opportunities endpoints
Implement the Skills and Categories lookup endpoints
Write unit and integration tests for the above modules
Write and maintain Swagger/OpenAPI documentation for all endpoints
Backend Dev 2 — Authentication, Services & Notifications
BE2 handles all security, business logic services, and the notification pipeline that the frontend and BE1 depend on.
Set up ASP.NET Identity with role-based access control (Job Seeker, Employer, Admin)
Build the full auth flow: register, login, logout, email verification, password reset
Implement JWT token generation, validation, and refresh token mechanism
Build the User Profile service (candidate and employer profile CRUD)
Build the Application service (business rules for applying, status transitions)
Build the Recommendation / Matching service (opportunity matching by skills)
Build the Notification service and email templates (SendGrid / SMTP)
Set up Hangfire background jobs for scheduled emails and alerts
Build the Dashboard API endpoints (stats for candidate, employer, admin)
Write unit tests for all services and auth flows
Frontend Dev 1 — Public Pages & Job Search
FE1 builds everything a visitor or job seeker sees before and during their job search — the public-facing experience of the platform.
Set up the Razor Pages project structure, shared layout, header, footer, and navigation
Build the Home page: featured opportunities, platform stats, call-to-action sections
Build the Opportunities Listing page: paginated cards, keyword search bar, filter panel
Build the Opportunity Detail page: full description, employer info, apply button, save button
Build the Candidate Browse page: searchable candidate cards with skills and location
Build the Candidate Profile Detail page: bio, skills, education, experience, resume download
Build the Employer Browse page: company cards with open positions count
Build the Employer Profile Detail page: company info, active listings
Build the Login and Registration pages with client-side validation
Build the Blog listing and Blog detail pages
Build the Contact and Pricing pages
Ensure all public pages are fully mobile-responsive using TailwindCSS
Frontend Dev 2 — Dashboards, Components & Admin Panel
FE2 builds the authenticated experience — all role-specific dashboards, the shared component library, and the admin panel.
Build the Candidate Dashboard: stats overview, applied jobs list, saved jobs grid, profile edit form
Build the Employer Dashboard: stats overview, manage jobs table, job posting form, applications inbox
Build the Admin Dashboard: user management table, content moderation, system statistics
Build the shared UI component library: job cards, candidate cards, status badges, modals, alerts, pagination
Build the Dashboard sidebar navigation with role-based menu items
Build the profile picture and resume upload UI flows
Build the rich-text job description editor in the employer job posting form
Implement all in-app notification UI: notification bell, notification centre, read/unread states
Implement client-side form validation across all dashboard forms
Ensure all dashboard pages are fully mobile-responsive with collapsible sidebar

4. How the Team Works Together
4.1 Shared Contracts (API First)
BE1 and BE2 write and publish Swagger API docs before FE1/FE2 start wiring data. This means frontend developers can build with mock/stub data immediately and swap in real endpoints as they become available.
BE1 publishes the Opportunities and Applications API contract by end of Week 2
BE2 publishes the Auth and User Profile API contract by end of Week 2
Both FE developers use Swagger UI to test endpoints independently
4.2 Git Branching Strategy
main
Stable, deployable code only — merged after review
develop
Active integration branch — all features merge here first
feature/be1-*
BE1 feature branches (e.g. feature/be1-opportunities-api)
feature/be2-*
BE2 feature branches (e.g. feature/be2-auth-service)
feature/fe1-*
FE1 feature branches (e.g. feature/fe1-job-listing-page)
feature/fe2-*
FE2 feature branches (e.g. feature/fe2-employer-dashboard)
4.3 Daily & Weekly Rhythm
Daily 10-minute standup: What did you finish? What are you working on today? Any blockers?
End of each week: each developer demos their completed work to the team
BE1 and BE2 sync at least twice per week to coordinate shared database changes
FE1 and FE2 sync at least twice per week to keep layouts and components consistent
All four meet at the start of each new week to confirm priorities and dependencies
4.4 Dependency Management
Some frontend work depends on backend APIs being ready. The table below shows the key dependencies to watch:
FE1 needs from BE1
Opportunities list & detail endpoints (Week 3)
FE1 needs from BE2
Auth endpoints — login/register (Week 2)
FE2 needs from BE1
Dashboard stats endpoints (Week 4)
FE2 needs from BE2
User profile endpoints, notification data (Week 4–5)
BE2 needs from BE1
DbContext, entity models, and repositories (Week 1–2)
4.5 Code Review Rules
No one merges their own pull request — always get at least one review
Backend PRs should be reviewed by the other backend developer
Frontend PRs should be reviewed by the other frontend developer
For any changes touching shared code (DbContext, layouts), all four review
